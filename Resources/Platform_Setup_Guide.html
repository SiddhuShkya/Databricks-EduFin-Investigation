01. Databricks Free Edition — Sign Up

You will work on the same platform used by production data engineering teams. Not a simplified playground — the real thing.

No credit card required. No cloud provider account. No trial period that expires.

Step 1 — Go to the Signup Page

Visit databricks.com/try-databricks

You will see two options: Free Trial and Free Edition.

Step 2 — Select Free Edition

Click "Get started with Free Edition"

Do not select the AWS, Azure, or GCP trial options. Those require cloud provider accounts and credit cards. Free Edition is the option on the right. It reads "For personal, non-commercial use."

Step 3 — Create Your Account

Sign up with Google, Microsoft, or email OTP. No password is required if using Google or Microsoft. Fill in your name and details — Databricks will create your workspace automatically.

Step 4 — Bookmark Your Workspace

Once your workspace opens, bookmark the URL immediately.

It will resemble: https://dbc-xxxxxxxx.cloud.databricks.com

This is your workspace. You will return to it every day.

What is included with Free Edition:
  • One workspace with serverless compute (auto-managed, no cluster setup)
  • One SQL warehouse (2X-Small, starts automatically)
  • Unity Catalog for data governance
  • Delta Lake tables
  • SQL Editor and Notebooks
  • Up to 5 concurrent job tasks
Known Issue: Some accounts may show "Cannot Create Serverless Cluster" immediately after signup. If this occurs, wait 30 minutes. If the issue persists, create a new account using a different email address. This resolves the problem.

02. Understanding Compute — What Runs Your Queries

When you write SQL and click Run, something must process that query. That "something" is compute. Databricks Free Edition provides two types.

Feature SQL Warehouse Serverless Notebook Compute
What it is A dedicated SQL engine optimized for analytical queries General-purpose compute that runs Python, SQL, Scala, and R in notebooks
Where you use it SQL Editor (sidebar icon) Notebooks
Best for Writing and running SQL queries during Days 1 through 5 Running the data generator notebook on Day 3
Startup time 2–6 seconds (fast) 10–30 seconds (cold start)
Free Edition limit One warehouse, 2X-Small size Limited to small cluster sizes
Auto-starts Yes, when you run a query in SQL Editor Yes, when you run a cell in a notebook
Auto-stops Yes, after idle timeout (conserves compute) Yes, after idle timeout
Simple rule for the week:

Days 1–2 and 4–5: Use the SQL Editor (runs on SQL Warehouse). Best for writing and executing SQL queries.

Day 3: Use a Notebook (runs on Serverless Compute). Required to run the Python generator that creates the 500,000-row dataset.
The 8-Minute Budget: Free Edition provides limited compute per day. One well-planned query is worth more than ten trial-and-error attempts. Write your SQL in a workbook first. Verify your logic. Run only when you are confident.

03. Uploading CSV Files (Days 1–2)

You receive 5 CSV files from Graphy. Each becomes a table inside Databricks.

Step 1 — Create the Schema

Open SQL Editor from the left sidebar (look for the SQL console icon). Run this command:

CREATE SCHEMA IF NOT EXISTS edufin_small;

This creates a container for your 5 tables. Think of a schema as a folder for related tables.

Step 2 — Upload Each CSV

In the left sidebar, click + New at the top, then click Data.

Select "Create or modify table". Drag and drop your first CSV file (for example, loans.csv) into the upload zone.

Step 3 — Configure the Table

Databricks will preview the data. Before clicking Create, confirm the following:

  1. Catalog: select workspace (or your default catalog)
  2. Schema: select edufin_small
  3. Table name: match the file name exactly:
    • loans.csv → table name: loans
    • customers.csv → table name: customers
    • dim_city.csv → table name: dim_city
    • dim_state.csv → table name: dim_state
    • institutions.csv → table name: institutions
  4. Confirm column types look correct (Databricks auto-detects these)
  5. Click Create

Step 4 — Repeat for All 5 Files

Upload each CSV file one at a time. All 5 must go into the edufin_small schema.

CSV FileTable NameExpected Rows
dim_state.csvdim_state18
dim_city.csvdim_city45
institutions.csvinstitutions50
customers.csvcustomers3,000
loans.csvloans5,000

Step 5 — Verify Your Upload

Open SQL Editor and run the following verification queries:

SELECT count(*) FROM edufin_small.loans; -- Expected: 5000 SELECT count(*) FROM edufin_small.customers; -- Expected: 3000 SELECT * FROM edufin_small.loans LIMIT 5;

If you see data returned, you are ready for Day 1.

Common mistake: Forgetting the schema prefix. SELECT * FROM loans will fail. Always use SELECT * FROM edufin_small.loans. This is how production databases work. Tables live inside schemas. Develop this habit from the start.

04. SQL Editor — Your Daily Workspace

The SQL Editor is where you will spend the majority of your time during this challenge.

Opening SQL Editor

Left sidebar → Click the SQL console icon (resembles a terminal prompt). A SQL warehouse will start automatically when you run your first query. Allow 2–6 seconds for startup.

Organizing Your Work

Create one query tab per phase. Name them clearly:

  • Phase 1 — Portfolio Health
  • Phase 2 — Geographic Risk
  • Phase 3 — Customer Risk
  • Phase 4 — Institutional Risk
  • Phase 5 — Timeline Analysis

Queries auto-save. Save manually (Ctrl+S) after significant progress.

Understanding Your Schema

Before writing complex queries, confirm what you have available:

-- View all tables in your schema SHOW TABLES IN edufin_small; -- View columns and data types for any table DESCRIBE TABLE edufin_small.loans; DESCRIBE TABLE edufin_small.customers; DESCRIBE TABLE edufin_small.dim_city;

How Tables Connect — JOIN Reference

You have 5 separate tables. They connect through foreign keys as follows:

loans.customer_id → customers.customer_id customers.city_id → dim_city.city_id dim_city.state_id → dim_state.state_id loans.institution_id → institutions.institution_id

Most business questions will require joining two or more tables. Example:

-- Which cities have the most loans? SELECT dc.city_name, COUNT(*) as total_loans FROM edufin_small.loans l JOIN edufin_small.customers c ON l.customer_id = c.customer_id JOIN edufin_small.dim_city dc ON c.city_id = dc.city_id GROUP BY dc.city_name ORDER BY total_loans DESC LIMIT 10;

05. Day 3 — Scale Shift (Generator Notebook)

On Day 3, you receive a Python notebook that generates 500,000 rows of data. This is where the challenge changes in character.

Step 1 — Download the Generator Notebook

Download the .ipynb or .dbc file from Graphy.

Step 2 — Import into Databricks

In your workspace: Left sidebar → Workspace → Click your username folder → Right-click → Import. Select the downloaded file. It will appear as a notebook.

Step 3 — Run the Notebook

Open the notebook. It will auto-attach to serverless compute. Run all cells from top to bottom — either via Run All at the top, or cell by cell using Shift+Enter.

This takes approximately 5–10 minutes. Progress messages will appear as data is generated. When complete, a new schema named edufin_national will be created with 5 tables at full scale.

Step 4 — Verify the Scale Shift

Return to SQL Editor and run the following verification:

SELECT count(*) FROM edufin_national.loans; -- Expected: 500,000 SELECT count(*) FROM edufin_national.customers; -- Expected: 100,000
From Day 3 onwards: Change all your queries from edufin_small to edufin_national. The table names, columns, and JOINs remain identical. The behaviour changes completely at 100x scale.

06. Scale Discipline — Non-Negotiable

In a serverless environment, planning is more valuable than typing. These five rules govern your entire week on the platform.

Rule 01 The 8-Minute Budget

Free Edition provides limited serverless compute per day. One well-planned query is worth more than ten trial-and-error attempts. You have approximately 8 minutes of active compute available per day. Use it with intention.

Rule 02 Plan Before You Run

Write your SQL in the workbook first. Check your WHERE clauses and LIMITs. Paste and execute only once you are confident the query is correct. Engineers think first, then act.

Rule 03 Always Use LIMIT

SELECT * without a LIMIT on 500,000 rows will crash your browser rendering and consume your daily budget. Every exploratory query must end with LIMIT 100.

Rule 04 Serverless Cold Starts

The 10–15 second delay on your first query is normal. The compute is initializing. Do not cancel and re-run. Doing so wastes multiple query starts. Wait for the result.

Rule 05 Schema Prefix Always

Always include the schema name. SELECT * FROM loans will fail. SELECT * FROM edufin_small.loans works. This is how production systems behave. Tables live in schemas — not in isolation.

Students who follow these 5 rules complete all phases comfortably within Free Edition limits.
Students who disregard them run out of compute by Wednesday.

07. Backup Platform — Google Colab + DuckDB

If Databricks compute runs out or encounters issues on any given day, switch to this platform immediately. Do not spend time debugging platform failures. Continue solving.

What is DuckDB?

DuckDB is a fast, in-memory analytical SQL engine. It runs inside Google Colab (free, browser-based) and reads CSV files directly with SQL. No server, no installation, no account is required beyond a Google login.

It supports all the SQL constructs required for this challenge: JOINs, GROUP BY, CTEs, window functions (LAG, LEAD, ROW_NUMBER), subqueries, and HAVING.

Step 1 — Open Google Colab

Navigate to colab.research.google.com

Click + New Notebook

Step 2 — Install DuckDB

In the first cell, run:

!pip install duckdb

Step 3 — Upload Your CSV Files

Click the folder icon in the left sidebar of Colab. Click the upload icon and upload all 5 CSV files. They will appear in the file browser as:

loans.csv, customers.csv, dim_city.csv, dim_state.csv, institutions.csv

Step 4 — Start Querying

In a new cell, run the following to confirm DuckDB is working:

import duckdb # DuckDB reads CSV files directly — no table creation required. duckdb.sql("SELECT * FROM 'loans.csv' LIMIT 5")

That is all. You are running SQL on your data.

Step 5 — Run Full Queries

Every SQL concept works in DuckDB. Here is an example using JOINs:

import duckdb result = duckdb.sql(""" SELECT dc.city_name, COUNT(*) as total_loans, SUM(CASE WHEN l.loan_status = 'Defaulted' THEN 1 ELSE 0 END) as defaults FROM 'loans.csv' l JOIN 'customers.csv' c ON l.customer_id = c.customer_id JOIN 'dim_city.csv' dc ON c.city_id = dc.city_id GROUP BY dc.city_name ORDER BY defaults DESC LIMIT 10 """) print(result)
Key difference from Databricks: In DuckDB, you query the CSV filename directly ('loans.csv') instead of a schema-qualified table name (edufin_small.loans). The SQL logic itself is identical.
When to switch to the backup platform:
  • Databricks shows "No compute resources available"
  • Queries hang for more than 2 minutes with no response
  • Your account shows persistent errors after multiple retries

Do not spend more than 10 minutes troubleshooting Databricks. Switch to DuckDB, continue solving, and return to Databricks the following day.

08. Troubleshooting

Reference this table when you encounter platform issues before escalating or seeking support.

Symptom Likely Cause Resolution
"Table not found" Missing schema prefix or incorrect table name Use edufin_small.loans not loans. Run SHOW TABLES IN edufin_small; to verify available tables.
"Schema not found" Schema has not been created yet Run CREATE SCHEMA IF NOT EXISTS edufin_small;
Query hangs (60s+) Cold start or missing LIMIT clause Wait 15 seconds. If still unresponsive, add LIMIT 100. Do not cancel and re-run.
"No compute resources" Daily compute budget exhausted Wait until tomorrow. Switch to DuckDB backup. Or create a new Databricks account.
Results show 0 rows Incorrect filter value or column name typo Run DESCRIBE TABLE edufin_small.loans; to verify exact column names.
Generator notebook fails Compute startup issue Wait 30 seconds and re-run the cells. If the issue persists, create a new Databricks workspace.
CSV upload fails File format issue or no active compute Ensure files end in .csv and have headers. Wait for the warehouse to start. Try uploading one file at a time.
"Cannot create serverless cluster" Known Free Edition provisioning bug Wait 30 minutes. If the issue persists, create a new workspace with a different email address.
Cannot locate SQL Editor UI navigation Left sidebar → Look for the SQL console icon (terminal prompt icon). This is distinct from the notebook icon.
